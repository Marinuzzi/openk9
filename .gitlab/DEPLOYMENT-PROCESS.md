## üåø Branch Types & Deployment Strategy

### **1. Feature Branches** (`123-feature-name`)
- **Pattern**: `^[0-9]+-.*$`
- **Deployment**: Single namespace based on user permissions
- **Image Tags**: Component-specific snapshots
- **Trigger**: Manual push by developer

### **2. Main Branch** (`main`)
- **Deployment**: Multiple namespaces based on component type
- **Image Tags**: Semantic versioning from `.version` files
- **Trigger**: Merge to main, manual push

### **3. Tags** (`v1.2.3`)
- **Deployment**: Production-ready, multiple namespaces
- **Image Tags**: Tag name
- **Trigger**: Git tag creation

## üöÄ **Main Branch Flow**

### **Scenario 3: Production deployment of Tika**

```mermaid
graph TD
    A[üîÄ Merge to main branch] --> B[üéØ OpenK9 Pipeline Triggered]
    B --> C{üîç Branch Detection}
    C --> D[üìã Main Branch Detected]
    D --> E[üèóÔ∏è Build Stage]
    E --> F[üìã Read .version file]
    F --> G[üîß tika-no-ocr built with tag 1.2.3]
    G --> H[üì¶ Image pushed to registry]
    H --> I[üîÑ Restart Template Triggered]
    I --> J[üåê Multiple External Pipeline Calls]
    J --> K[üéØ k9-backend: ArgoCD Update]
    J --> L[üéØ k9-backend01: ArgoCD Update]
    J --> M[üéØ k9-test: ArgoCD Update]
    K --> N[üîÑ Restart in k9-backend]
    L --> O[üîÑ Restart in k9-backend01]
    M --> P[üîÑ Restart in k9-test]
    N --> Q[‚úÖ Production Ready]
    O --> Q
    P --> Q
```

**Multi-Environment Deployment:**
1. **Trigger**: Code merged to `main` branch
2. **Versioning**: Reads version from `.version` file (e.g., `1.2.3`)
3. **Build**: Creates production image with semantic version
4. **Multi-Call**: Template makes **3 separate calls** to external pipeline:
   ```bash
   # Call 1: k9-backend
   curl -X POST -F "variables[NAMESPACE]=k9-backend" ...
   
   # Call 2: k9-backend01  
   curl -X POST -F "variables[NAMESPACE]=k9-backend01" ...
   
   # Call 3: k9-test
   curl -X POST -F "variables[NAMESPACE]=k9-test" ...
   ```
5. **Parallel Deployment**: Each call updates the respective namespace
6. **Result**: `tika-no-ocr:1.2.3` running in all backend environments

## üîç **Pipeline Decision Logic**

### **User Permission Validation**
```bash
# Feature Branch Logic
if [[ "$CI_COMMIT_BRANCH" =~ ^[0-9]+-.*$ ]]; then
  case "$OPENK9_USER" in
    "mirko.zizzari")
      case "$COMPONENT_TYPE" in
        "backend") NAMESPACE="k9-backend" ;;
        "backend-oracle") NAMESPACE="k9-backend-oracle" ;;
        "test") NAMESPACE="k9-test" ;;
      esac ;;
    "lorenzo.venneri"|"giorgio.bartolomeo")
      if [[ "$COMPONENT_TYPE" == "frontend" ]]; then
        NAMESPACE="k9-frontend"
      else
        echo "‚ö†Ô∏è Frontend developers can only deploy frontend components"
        exit 1
      fi ;;
  esac
fi
```

### **Component Type Detection**
```bash
# Fallback logic when COMPONENT_TYPE not specified
case "$COMPONENT_NAME" in
  # Frontend components
  *-ui|admin-ui|tenant-ui|search-frontend)
    NAMESPACES="k9-frontend" ;;
  # Backend components  
  datasource|searcher|ingestion|tika-no-ocr|entity-manager)
    NAMESPACES="k9-backend k9-backend01 k9-test" ;;
  # AI components
  *-ai|embedding-*|rag-*)
    NAMESPACES="k9-ai" ;;
esac
```

## üö® **Error Scenarios & Troubleshooting**

### **Common Issues**

#### **1. Permission Denied**
```bash
‚ö†Ô∏è User luca.callocchia can only deploy 'ai' components, got 'backend'
```
**Solution**: User tried to deploy wrong component type

#### **2. App Not Found**
```bash
‚ùå App k9-backend/unknown-component not found!
üí° Available apps in namespace k9-backend:
   k9-backend/datasource
   k9-backend/tika-no-ocr
```
**Solution**: Component doesn't exist in target namespace

#### **3. Unknown User**
```bash
‚õî Unknown OpenK9 user 'new.developer' for feature branch deployment
üí° Available users: mirko.zizzari, michele.bastianelli, luca.callocchia...
```
**Solution**: Add user to pipeline configuration

## üéØ **Best Practices**

### **For Developers**
1. **üåø Use descriptive branch names**: `123-fix-tika-memory-leak`
2. **üß™ Test in your namespace first**: Feature branches deploy to your dev environment
3. **üìù Update version files**: Before merging to main
4. **üîç Check pipeline logs**: Verify successful deployment

### **For DevOps**
1. **üë• Manage user permissions**: Update pipeline when team changes
2. **üè∑Ô∏è Monitor image tags**: Ensure proper versioning
3. **üìä Track deployments**: Use ArgoCD UI for deployment status
4. **üö® Handle failures gracefully**: Partial failures in multi-namespace deployments

### **For Releases**
1. **üè∑Ô∏è Create semantic tags**: `v1.2.3`, `v2.0.0-rc1`
2. **üìã Update changelogs**: Document changes
3. **üß™ Test in staging**: Use `k9-test` namespace
4. **üì¢ Communicate deployments**: Notify team of production changes
